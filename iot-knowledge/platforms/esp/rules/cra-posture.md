# CRA Secure-by-Design Posture — ESP (platforms/esp/rules/cra-posture.md)

The secure-by-design **posture preview** for ESP-IDF. Parallel to the nRF posture, reading ESP's config
(`sdkconfig`). Each check = real evidence → status → **plain-English requirement** → suggested action.
Dated **"11 Dec 2027 essential requirements — worth doing now."**

## Overview
- **Read the project `sdkconfig`** (or `build/config/sdkconfig.json`), not a guess. It already reflects
  the merged defaults + `sdkconfig.defaults` + menuconfig. If absent, say so and offer a build.
- **Honest statuses only:** ✅ = "configured/present" · ⚠️ = "review" · ❌ = "not found". Chip-dependent
  checks that don't apply are **N/A**, never a fail. Flash encryption / secure boot are **production-mode**
  concerns — flag "development mode" enablement as ⚠️, not ✅.

> **Requirement citations are plain-English in v0.1.** Exact CRA **Annex I clause letters** + **EN 18031**
> per row are a **fast-follow pending expert validation** — do not assert clause letters yet.

## The checks (evidence → status → requirement → action)

| # | Check | Evidence (`sdkconfig`) | Plain-English CRA requirement | If missing → suggested action |
|---|---|---|---|---|
| 1 | **Secure boot v2** | `CONFIG_SECURE_BOOT=y` (+ `CONFIG_SECURE_BOOT_V2_ENABLED`) | Boot only signed, verified firmware (root of trust). | Enable Secure Boot v2 (the root — do **first**; production keys handled carefully). |
| 2 | **Flash encryption** | `CONFIG_SECURE_FLASH_ENC_ENABLED=y` (release mode) | Protect firmware/data confidentiality at rest. | Enable flash encryption in **release** mode (⚠️ if only development mode). |
| 3 | **Signed OTA / rollback** | `CONFIG_APP_ROLLBACK_ENABLE`, OTA + signed-image verify | Ship integrity-protected updates over the support period. | Enable OTA with rollback **after** secure boot (signed images need it). |
| 4 | **Transport security (TLS)** | `CONFIG_ESP_TLS_*` / `CONFIG_MBEDTLS_*`; flag `*_SKIP_*`/insecure | Encrypt + authenticate network traffic; no cert-skip in production. | Use esp-tls with proper cert verification; remove insecure skips. |
| 5 | **NVS encryption** | `CONFIG_NVS_ENCRYPTION=y` | Protect stored secrets/keys/credentials. | Enable NVS encryption (depends on flash encryption). |
| 6 | **JTAG / download mode locked** | secure boot disables ROM DL mode; `CONFIG_SECURE_DISABLE_ROM_DL_MODE` | Limit the debug/attack surface in production. | Disable ROM download mode / JTAG for production. |
| 7 | **Memory / stack protection** | `CONFIG_COMPILER_STACK_CHECK*`, `CONFIG_ESP_SYSTEM_MEMPROT*` | Mitigate memory-corruption exploits. | Enable stack-check + memory protection where supported. |
| 8 | **Logging hygiene** | log level config + heuristic grep of `main/`/`src/` for secrets in `ESP_LOG*`/`printf` | Don't leak secrets; minimise sensitive data in logs. | ⚠️ review flagged lines (heuristic — confirm manually). |
| 9 | **Console in production** | `CONFIG_ESP_CONSOLE_UART*` enabled in a production build → flag (`CONFIG_ESP_CONSOLE_NONE` = off) | Don't ship an interactive debug console in production. | Set console to none for production builds. |
| 10 | **SBOM generated** | this run produced `compliance/sbom/` | Identify & document components (machine-readable SBOM). | Generated by the SBOM step (`esp-idf-sbom`). |

## "Do these first" — dependency order
1. **Secure Boot v2** (root). 2. **Flash encryption** (release). 3. **NVS encryption** (needs flash enc).
4. **Signed OTA + rollback**. 5. **TLS hardening**, JTAG/DL-mode lock, memory protection, logging, console-off.

> Verification status: ESP Kconfig→requirement mappings are best-effort and pending the same expert pass
> as nRF. Treat as a *preview*, not an audit. (Reuses the cross-vendor SBOM; this brings the **posture**
> to ESP so an ESP user is never left with an empty table.)
